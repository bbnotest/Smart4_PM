# 🟧 Jenkins · 3단계 — 빌드 읽기 (성공·실패·로그)

> 🎯 **개요** — 자동 빌드의 진짜 가치는 **"결과를 읽는 것"**에 있습니다. 성공/실패가 어디에 보이고, 실패하면 **왜인지 로그에서 찾는** 법을 익힙니다. PM·QA의 일상 기술이에요.

## A. 빌드 기록 & 공 색깔 읽기

Job 화면 좌측 **`Build History`**에 빌드가 `#1`, `#2`… 쌓입니다. 각 빌드 앞 **색깔 공**의 뜻:

| 표시 | 뜻 |
|---|---|
| 🔵/🟢 **파랑·초록 공** | **성공**(SUCCESS) |
| 🔴 **빨간 공** | **실패**(FAILURE) — 명령이 오류로 끝남 |
| ⚪ **회색 공** | 실행 안 됨/취소(중단) |
| 🔆 **깜빡이는 공** | 지금 **빌드 진행 중** |
| ⛅ **날씨 아이콘** | 최근 빌드들을 종합한 **안정도**(맑음=대체로 성공) |

> 🙋 Jenkins는 전통적으로 성공을 **파란 공**으로 표시합니다(역사적 이유). 요즘 테마·플러그인에선 **초록**으로 보이기도 해요. 색이 무엇이든 **파랑/초록=성공, 빨강=실패**로 기억하면 됩니다.

## B. 콘솔 출력에서 원인 찾기

1. 실패한 빌드 번호(예: `#2`) 클릭 → **`Console Output`**(콘솔 출력)
2. **맨 아래부터** 읽습니다 → `Finished: FAILURE` 바로 위에 보통 원인이 있어요
3. `error`, `not found`, 빨간 글자 근처가 단서입니다

> 🔸 핵심 습관: **"빌드 깨졌다 → 추측하지 말고 로그부터."** 콘솔 맨 아래 몇 줄이 거의 항상 답을 줍니다.

## C. 일부러 실패시켜 보기 (두려움 없애기)

실패 로그를 한 번 만들어 보면 무서움이 사라집니다.

1. Job → `구성`(Configure) → 빌드 단계의 명령을 아래로 바꾸기:
   ```bat
   echo 일부러 실패 테스트
   exit 1
   ```
   (`exit 1` = "오류로 끝났다"는 신호)
2. `Save` → `Build Now` → **빨간 공** 등장 → `Console Output`에 `Finished: FAILURE` 확인
3. 확인했으면 명령을 **2단계의 원래 echo로 되돌리고** 다시 빌드 → 다시 성공 공

> 🙋 실패는 잘못이 아니라 **정보**입니다. 자동 빌드의 목적이 바로 "문제를 **빨리** 빨간불로 알려주는 것"이에요.

## D. 작업공간(Workspace) = 빌드가 일어나는 폴더

좌측 **`Workspace`**(작업 공간)는 Jenkins가 실제로 파일을 만드는 폴더입니다. 2단계에서 만든 `result.txt`가 여기 있어요. (5단계에서는 여기서 Unity가 만든 `.exe`를 보게 됩니다.)

## 🎮 현장 감각 — 게임 PM은 이렇게

> **Pixel Dungeon 맥락**<br>
> 빌드 상태판은 팀의 **신호등**입니다 — 빨간 공이면 "지금 합치면 안 돼요"라는 뜻.<br>
> PM은 코드를 못 고쳐도, **"언제부터 빨간불인지, 무엇이 원인 줄인지"**를 로그로 짚어 담당자에게 넘길 수 있어야 합니다.<br>
> "무에러"는 화면만 보고 단정하지 말고 **콘솔 로그로 확인**하는 습관, 이게 실무의 기본기예요.

**⚠️ 흔한 실수**
- 빨간 공 보고 패닉 → **콘솔 맨 아래부터** 읽으면 대부분 원인이 보입니다.
- 성공인데 결과물이 없음 → **Workspace**를 확인(경로·파일명 문제).

**🎤 면접 한 줄**
> *"빌드 **성공/실패를 콘솔 로그로 판독**하고, 실패 원인 줄을 짚어 담당자에게 전달하는 흐름을 익혔습니다."*

## ✅ 확인

- [ ] 공 색깔(파랑·초록=성공/빨강=실패)을 구분할 수 있다
- [ ] `Console Output`에서 `Finished: SUCCESS`/`FAILURE`를 찾을 수 있다
- [ ] 일부러 실패(`exit 1`)시켜 빨간 공을 본 뒤 다시 성공으로 되돌렸다
